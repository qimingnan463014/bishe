# DEV_PLAN

> 用途：记录“视频 vs 当前程序”的差异清单，并在修复后更新 `[ ]` / `[x]`。

## 差异清单（待修复）

1. [x] **紧急修复**：全局修复 `year_month` (MySQL保留关键字) 导致的 500 SQLSyntaxErrorException
   - 现状：各类基于 `year_month` 过滤的列表/统计接口报错。
   - 实施子任务：
     - [x] Java实体层：所有含 `yearMonth` 属性的 Entity (`SalaryRecord`, `Performance`, `AttendanceRecord` 等) 强制加上 `@TableField("\`year_month\`")` 注解。
     - [x] XML映射层：所有手写 SQL (`*Mapper.xml`) 中的 `year_month` 全部替换为 `` `year_month` ``以转义。
     - [x] Wrapper构造器：排查 `QueryWrapper` 和代码中直接传入字面量 `"year_month"` 的场景，保证与带反引号的命名完全对齐。
2. [x] 薪资“二审/已审核(3)”流程缺失导致支付门控不可靠
   - 现状：`t_salary_record.calc_status` 语义包含 `3=已审核`，但业务只把发布置为 `2=待审核`，没有二审把状态置为 `3`；支付允许 `calcStatus>=2`。
   - 影响：可能绕过审核直接发放。
   - 实施子任务：
     - [x] 后端：新增 `/audit` 二审接口，补齐 `2 -> 3` 的状态转换。
     - [x] 后端：修改发钱接口 `paySalary`，拒绝 `calcStatus != 3` 的非合规请求，彻底堵死漏洞。
     - [x] 前端：新增高逼格弹窗版“模拟聚合支付收银台”（含微信/支付宝/银行网银假交互及成功动效）。
     - [x] 前端：补充列表顶部的【批量发放】按钮，打通并列验证门控。
2. [ ] 薪资反馈缺少流程联动（无法影响审核/阻塞发放）
   - 现状：`SalaryFeedbackServiceImpl.replyFeedback()` 仅更新反馈表状态与回复内容，不联动薪资账单状态与二审闭环。
   - 影响：反馈提交后系统仍可能直接进入发放流程，且“二审依据”缺失。
   - 相关代码：`SalaryFeedbackController`、`SalaryFeedbackServiceImpl`、二审接口（需要补齐）。
3. [ ] 薪资发放“发放文件 issueFile”字段链路缺失
   - 现状：前端“薪资发放”表格展示 `issueFile` 列，但 `SalaryPayment` 实体/`t_salary_payment` 表结构未见该字段。
   - 影响：视频里那种“发放文件/凭证”列无法落地展示。
   - 相关代码：DB（`t_salary_payment`）、`SalaryPayment` 实体、`SalaryPaymentServiceImpl`/发放逻辑、前端 `admin/index.html`。
4. [ ] 后端鉴权/状态校验不够一致（敏感操作可能被绕过）
   - 现状：前端按钮做了权限与状态隐藏，但后端在部分接口上缺少严格 role 校验与状态门控（避免“改请求直接操作”）。
   - 影响：安全性与业务一致性不足。
   - 相关代码：`SalaryController`、`SalaryFeedbackController`、`SalaryPaymentController` 以及其它涉及敏感操作的 controller/service。
5. [x] 全局 UI 统一化：操作按钮与表格分页组件规范化
   - 现状：部分表格缺失底层分页组件，操作按钮（详情、修改、删除等）颜色散乱无统一风格。
   - 影响：系统割裂感强，多组件切换显得不专业且容易引发多 Agent 修改造成的二次回归。
   - 实施：通过自动扫描注入 `p` 响应式状态实现纯前端切片分页，利用正则规范化 `primary/warning/danger/success` 的语义色值。
6. [ ] 员工侧完整流程页面（薪资详情/历史/反馈）可能存在占位未实现
   - 现状：目前仓库中的员工入口页面（如 `static/index.html`）大量模块是 Placeholder/TODO。
   - 影响：与视频展示的全流程不一致。
   - 相关代码：`salary-system/src/main/resources/static/index.html`（或其它员工前端资源）。

> 说明：详细“变更记录”已迁移到 `CHANGELOG.md`。

7. [x] 管理员端导出格式修复与算薪交互优化
   - 现状：薪资核算需强选月份交互不友好，各大表格只支持CSV导出。
   - 修复：增加当月缺省逻辑，全表导出切为 xlsx 格式。

8. [ ] 考勤自动化处理（打卡边界识别）
   - 现状：当前只是手工/直接填入迟到次数和加班小时数。
   - 目标：按照论文更改要求，支持通过导入包含打卡时间（09:00、18:00、19:00边界）的考勤源文件，自动算出本月迟到、早退（各扣50元）、加班（每满一小时加20元）的绝对值并注入基础数据。

9. [x] 绩效总分自动化评级与奖金直接挂钩 (四重累加分算法)
   - 现状：三项平均分算比例旧版逻辑，奖金按基数乘积比例发放，存在耦合且灵活性差。
   - 目标实现：废除“三项平均+奖罚”公式。采用“工作绩效(50) + 业务技能(10) + 工作态度(10) + 员工考勤(30)”的绝对值累加。
   - 评级调整：<=60(不合格/无), 61-75(合格), 76-85(良好), 86-100(优秀)。金额前端控制，映射后以绝对值落库至 `perfBonusRatio` 字段并在薪资核算时直接提取。

10. [x] 五险一金依据“基本工资”正向计提及失业保险展示
   - 现状：旧版由于后端直接通过 `17.5%` 返回扣划总额，导致前端明细受限及计算逻辑绕弯。
   - 目标实现：在前端直接依靠 `row.baseSalary` 提取：“养老(8%)、公积金(7%)、医疗(2%)、失业(0.5%新增展示列)”，不仅补充了失业保险列，还修复了经理账号和姓名的缺省显示。

11. [ ] 部门经理“一键上报/汇总”闭环交互
   - 现状：经理虽然有批量计算的能力，但在UI上没有一键整体发布的显性闭环链路或总览报表。
   - 目标：允许经理通过“一键确认发布”将部门下所有人薪水直接推送上报。管理员拥有最终支付权。
